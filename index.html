Android屏幕适配

發表於 2015-09-23

  1. Android屏幕适配分辨率

    常见需要适配的屏幕分辨率

    • 1280 x 720
    • 800 x 480
    • 854 x 480
    • 1920 x 1080
    • 960 x 540
    • 480 x 320

  2. 密度无关像素：dot per inch(dpi)

    与屏幕尺寸和屏幕分辨率有关，以Nexus 5为例.
    屏幕分辨率1920x1080，对角分辨率P = sqrt( 1920^2 + 1080^2 ) = 2202.9。使用对角分辨率除以屏幕尺寸4.95英寸即可得到密度无关像素为445dpi.


    以密度无关像素对设备进行区分
    mdpi ( x 1) —— 120 ～160 dpi
    hdpi ( x1.5) —— 160 ～240 dpi
    xhdpi( x 2) —— 240 ～320 dpi
    xxhdpi(x 3) —— 320 ~ 480 dpi
    xxxhdpi(x 4) —— 480 ~ 640 dpi
    图片大小按照括号内的大小进行放缩，计算出机型的dpi以后对应到指定的类型中去

  3. 文字大小

    推荐使用12sp以上的字体大小，并且推荐大小为12/14/18/22sp的字体，并且尽量不要使用奇数和小数，其容易在放大或缩小过程中导致精度丢失。

  4. 限定符（尺寸限定符和方向限定符可以连着使用）
  • 尺寸限定符
    创建不同的文件夹，在不同的文件夹下创建相同的文件，使用不同的方式进行应用
    res/layout/main.xml —— 常规文件
    res/layout-large/main.xml —— Android 3.2版本前平板等设备使用的layout
    res/layout-sw600dp/main.xml —— Android 3.2版本以后等设备使用的layout，指最小宽度大于600dp的设备（一般即指平板）

小技巧：
对于以上情况可以使用布局别名
创建两个文件：
res/layout/main_vertical.xml —— 竖直状态布局
res/layout/main_oriental.xml ——水平状态布局

默认布局：res/values/layout.xml

Android 3.2版本前的布局：res/values-large/layouts.xml

Android 3.2版本后的布局：res/values-sw600dp/layouts.xml

在对应的Activity中只需要setContentView( R.layout.itemname)即可

  • 方向限定符
    land / port
    与尺寸限定符类似，创建对应的layouts.xml文件
    res/values-land/layouts.xml
    res/values-port/layouts.xml
    之后同上创建标签来指定对应的布局文件

  • 9-patch图
    左侧和上侧用于标记可拉伸的位置，实际拉伸由点的数量按照比例进行拉伸。
    右侧和下侧用于标记，未被标记的区段将会被设置成padding值

5.按照百分比进行布局：
Android最新的百分比布局库：android-percent-support-lib，可以使用百分比的方式进行Layout布局，主要支持PercentRelativeLayout和PercentFrameLayout
支持的属性有：
layout_widthPercent、layout_heightPercent
layout_marginPercent、layout_marginLeftPercent、layout_marginRightPercent等6个属性

使用： compile ‘com.android.support:percent:22.2.0’

android Ultra-Pull-To-Refresh项目分析

發表於 2015-09-18   |   分類於 编程学习

dispatchTouchEvent()函数内部：
Action_Down:

  • 自己的处理步骤：
  1. 初始化mHasSendCancelEvent 为false，表示此时PtrFrameLayout未发送ACTION_CANCEL，此时的ACTION动作由PtrFrameLayout的子View来响应。
  2. 设置Indicator状态：mIsUnderTouch 为true，保存当前按下的位置为mPtLastMove，mPressedPos设置为mCurrentPos。
  3. 查看ScrollChecker的状态，如果正在执行则终止任务。
  • 分发给子View：
    ACTION_DOWN事件仍将分发给其内部的子View，从而子View也可以响应其他事件，而在当父控件开始响应到下拉操作时，PtrFrameLayout将会发送一个ACTION_CANCEL事件打断内部View对后续事件的响应。

Action_Move：

  • 在PtrIndicator中保存滑动的逻辑和状态

  • 判断不执行下拉刷新的情况

  1. 检查是否设置水平移动禁用模式

    通过disableWhenHorizontalMove(boolean disable)函数来设置，主要用于当子View中存在水平监听响应的控件（如ViewPager）时，启动该选项，那么此时则会检测当前的位移情况，如果此时PtrFrameLayout是未滑动的状态，此时不执行下拉刷新，而通过父类的dispatchTouchEvent()将事件传递给mContentView的子类中。

  2. 检查下拉状态

    在移动状态为moveDown时，检测mPtrHandler的状态，调用checkCanDoRefresh()判断，如果此时是不可进行下拉刷新的，则也通过父类的dispatchTouchEvent()将事件传递给mContentView的子类中。

  • 如果以上两种状态均不满足，获取当前的运行状态

Android学习 —— RecyclerView的Adapter使用
前两天照着网上的内容写了一个嵌套的Adapter，当时觉得设计很巧妙，然而并没有什么卵用，感觉其实可以很简单的问题被复杂化了，后来仔细研究了一下，发现其实蛮简单的。

几个重要函数的重载

  1. public int getItemCount()
    个人理解来看，这个函数将会被首先调用，从而RecyclerView将会根据这个函数判断需要创建多少个ViewItem，一般都是返回Adapter对应内容的List的size()

  2. public int getItemViewType(int position)
    接下来将根据获取的item数量逐个创建item。这个函数用来获取ListView在指定position的Type，从而系统或根据获取到的Type类型来决定是重用已经用过的ViewHolder还是重新新建一个ViewHolder。

  3. public RecyclerView.ViewHolder onCreateViewHolder(ViewGroup parent, int typeView)
    个人理解：此处parent就是之前在RecyclerView中创建的一个一个的item，而后根据前一个函数获取到的ViewType，从而加载不同的内容，并且将形成的整个视图作为ViewHolder返回出去。所以此处主要执行inflate函数，并且将结果view作为传给各个ViewHolder作为参数执行findViewById()，并将内容保存在ViewHolder的内部类里面。

  4. public void onBindViewHolder( RecyclerView.ViewHolder viewHolder, int position)
    该个函数将会根据上一个函数所创建的ViewHolder，并且将这个ViewHolder作为参数传递到这个函数中，所以可以根据参数中的position得到类型名称并且将获取到的ViewHolder的类型进行强制类型转换，并且将具体的View的内容进行设置，相当于在Fragment中调用getTag()函数以后进行的操作。

  5. 其他注意事项
    创建RecyclerView中的各个子类的ViewHolder，均继承自RecyclerView.ViewHolder。各个类均只包括其中的各个控件，并且重写构造函数，参数为View（其实就是parentView）。并且在构造的时候使用findViewById()将各个xml控件和java控件绑定。

Android LayoutAnimation

發表於 2015-08-18   |   分類於 编程学习

  • 在使用系统默认的LayoutAnimation时，只需要在Layout中添加属性android:animateLayoutChanges = “true” 即可。

  • 自定义LayoutAnimation

  1. 创建LayoutTransition对象

  2. 设定LayoutTransition的动画内容，包括

  • APPEARING—— 此View在Layout中出现时，此View需要执行的动画。
  • CHANGE_APPEADING——当一个View在Layout中出现时，此View对其他View位置造成影响，从而其他View需要执行的动画
  • DISAPPEARING—— 此View在Layout中消失时，此View需要执行的动画。
  • CHANGE_DISAPPEARING——当一个View在Layout中消失时，这个View将对其他View的位置造成影响，从而其他View需要执行的动画
    Ps: 在上文所述的默认情况下DISAPPEARING和CHANGE_DISAPPEARING都是立即开始，没有动画的。
