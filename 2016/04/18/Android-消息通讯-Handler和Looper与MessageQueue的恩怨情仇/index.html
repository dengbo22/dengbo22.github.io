Handler和Looper与MessageQueue的恩怨情仇

发表于 2016-04-18   |   分类于 编程笔记   |     |  

Message是Android中最为常用的消息通讯机制，这种机制是建立在Handler、Looper和MessageQueue的分工协作上。Message通过Handler分发到MessageQueue中，MessageQueue将所有的Message组织成一个链表，通过Looper不断的将MessageQueue中的Message取出，取出以后又交付给对应的Handler来处理，从而完成一整套消息通讯机制。

各个类的简介

  • Message

    作为这套机制的核心，躲不掉首先介绍Message，其本身Android的Parcelable序列化对象，其内部通过静态变量sPool实现了一个缓存池用来回收和创建Message

    从上我们可以看到，Message的构造函数并没有重载，其实一般情况下Message都不是通过构造函数来创建，而是从Message的静态成员sPool中重用已经创建过的Message对象，用完以后再将对象放回缓存池。从而降低反复创建Message的开销

  • Looper

    Looper是一个消息循环结构，其内部具有一个ThreadLocal变量，ThreadLocal是一个可以在每个线程中存储数据的数据存储类，多个线程通过同一个ThreadLocal获取到的东西是不一样的。Looper利用这个ThreadLocal变量，针对线程保存了Looper对象，从而使得一个线程中只有一个Looper对象。首先看一下Looper这个类的对象成员和构造函数。

    实质上，Looper为了保证一个线程只有一个Looper，将线程Looper对象存放到ThreadLocal对象中。因为不同的线程通过同一个ThreadLocal取得的对象是不一样的，所以不同的Thread获取到的Looper也是不同的，从而保证不同的线程获得不同的Looper。

    从上面我们可以看出来，Looper内部具有MessageQueue变量mQueue，在创建的时候，自身跟当前线程进行绑定。从而对于任何一个线程都可以通过静态方法myLoop()获取当前线程中的Looper对象。

  • Handler

    Handler其实质上相当于一个搬运工，其不仅将一个个集装箱（Message）放到传送带（Looper）中，同时也会将集装箱（Message）从传送带上取下来进行处理和操作。Handler实质上是和Looper绑定的，但是这种绑定是通过Thread来进行绑定的，也就是说，哪个线程创建了Handler，那么Handler就会和这个Handler绑定，从而操作这个Thread中的Looper和Looper中的MessageQueue。

发起消息

一般来说，一个消息通讯首先是从发起消息开始的，一般来说，我们将通过Handler的sendMessage或者通过其Post方法发送消息

从上我们可以看出，不论是sendMessage()还是post()，最后实质上都会调用sendMessageDelayed()方法，那么继续往下看。

看到最后将会调用MessageQueue对象的enqueueMessage()，将信息存入MessageQueue中

这样，消息就被存放到了MessageQueue中。总结一下，不论调用哪种方式进行消息发送，都会sendMessageDelayed函数，而这个函数通过层层的调用最终将会获取当前线程的MessageQueue，并调用MessageQueue的enqueueMessage函数，将消息通过传入的参数when的值来确定Message应该在MessageQueue的具体位置，最终将Message插入MessageQueue的合适位置中。规则是，when的时间越小，其在MessageQueue中的位置就越靠前。
那么，消息又是如何一个一个被取出来的呢，这个就得交给Looper来做消息取出了。

消息取出

前面说到Looper是一个循环器，当其调用loop()以后将会不断的循环从Loop中的MessageQueue中取得对象

我们可以看到，queue.next()将会从MessageQueue中取得一个Message，这个Message的target的对象实质上是一个Handler，所以实质上是由Message设定的目标Handler来调用dispatchMessage()。

总结一下，收取信息Looper调用loop()函数执行循环，不断的调用MessageQueue中的next()来检查是否是合适的时间获取Message，获取到信息以后，使用Message的target来处理该Message，处理的优先顺序为：优先Message自身设置的回调，之后是Handler设置的回调，如果都没有，则最后才使用handler中定义的handlerMessage方法来处理该Message。

至此，一个消息的旅途结束。
